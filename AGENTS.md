# 项目协作规则

- 所有回复一律使用中文。
- 使用中文内容时, 统一使用半角标点符号.
- 修改代码中已有的注释（包括函数注释、行内注释、文件头注释）时，必须逐条列出原注释、建议修改内容和可选处理方案，由用户选择确认后才能写入文件。
- 检索或阅读代码时发现已有注释与当前实现不一致、语义过期或存在疑问时，必须主动列出问题注释并与用户讨论，确认后再修改。
- 新增注释无需确认，可直接写入。
- 修改代码前，先简要说明将要修改的文件和目的。
- 优先遵循项目现有代码风格、目录结构和命名习惯。
- 避免过度设计, 保持实现简单直接; 如果发现更好的设计方式, 先说明方案/收益/风险/取舍, 与用户讨论确认后再实施.
- 尽早暴露错误和失败, 不要使用不符合既定流程的修补数据或临时流程来掩饰问题; 需要临时绕过时, 必须说明原因、影响范围和恢复方案并征求确认.
- 遇到任何疑问、不确定点，或发现可能存在更好方案时，先征求用户确认。
- 所有文件编码统一采用 UTF-8。
- 命令工具统一使用 Git Bash。
- 涉及依赖安装、联网下载、权限提升或可能影响本机环境的操作时，先征求用户确认。

# 文档同步约定
- 所有代码修改都必须同步更新对应服务目录下的 `README.md` 设计文档，保证设计文档与当前代码实现一致。
- `README.md` 至少应覆盖服务设计、架构、核心能力、数据流、接口、注意事项、待改进项等内容；涉及行为、依赖、配置、部署或接口变化时，需要同步补充或修正对应章节。
- 修改代码、proto、配置、部署文件或启动流程时，必须同步检查相关 README、TEST、部署说明和配置模板是否过期。
- 如果相关文档需要更新，必须在同一轮改动中一起更新。
- 如果涉及修改已有注释或已有文档说明，必须先列出拟修改内容并征求用户确认；确认后再写入文件。
- 如果确认无需更新文档，最终回复中必须说明已检查哪些文档以及为什么无需修改。
- 任何涉及 login/gateway/online/cache 行为、接口、配置、部署或测试流程的改动，都必须检查对应目录下的 `README.md`、`TEST.md`、`deploy/*/README.md`、`bin/*.yaml.template` 和相关 `deploy/*.yaml`，并同步更新过期内容。

# 代码风格
- 生成或修改业务代码时，优先检查并使用 `github.com/75912001/xlib` 中已有模块，避免重复实现通用能力。
- 常见优先复用模块包括但不限于 `xlib/map`、`xlib/timer`、`xlib/actor`、`xlib/control`、`xlib/etcd`、`xlib/grpc`、`xlib/log`、`xlib/runtime`。
- 仅当 `xlib` 没有合适能力、标准库实现更直接，或当前场景需要特殊处理时，才新增本地实现。
- 新增本地通用实现时，需要简要说明未使用 `xlib` 的原因。
- 外部输入 / 跨模块边界 / 状态变更函数: 函数本身必须校验参数合法性.
- 内部小函数 / 已校验链路 / 性能敏感路径: 由调用者保证参数合法性, 函数可只做必要防护.
- 确认没有必要的代码, 不需要保留.
- 设计流程顺序时, 先执行本地纯计算和本地状态检查, 再执行 cache/gRPC/Redis/etcd 等外部副作用操作.
- 不产生外部副作用的检查必须尽量前置, 避免先写外部状态后再因为本地条件失败而额外回滚.
- 写 cache、绑定 online、注册服务、更新远端状态等外部副作用应尽量靠后; 一旦开始外部副作用, 后续失败才进入显式回滚/补偿区域.

# 注释规则
- 中文注释优先，必要时可添加英文注释以补充说明
- 生成代码时，需要为关键行为、重要变量、特殊用法和注意事项添加必要注释。

- 不要执行破坏性命令，例如 `git reset --hard`、强制删除文件或大范围重写历史，除非用户明确要求。
